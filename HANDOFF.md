## Decisions Made

- **Auth:** no real auth. `app/dependencies.py:get_current_user_id` reads `X-User-Id` header, falls back to a stub UUID (`00000000-0000-0000-0000-000000000001`). User explicitly chose "stub/fixed user" over header-required and over body-param approaches.
- **Scope:** Parts 1–5 only. Part 6 (custom exceptions, pagination, task history) deferred. Service raises `HTTPException` directly (same compromise `UserService` makes).
- **`create_task` with assignee routes through domain:** if `data.assignee_id` is set, call `task.assign_to(user)` inside the service so the inactive-user invariant still fires on creation.
- **Migration approach:** autogen-based. User chose to regenerate via `alembic revision --autogenerate` over keeping the patched hand-written `0001_initial.py`. Autogen file has known divergences — see below — not yet patched.
- **docker-compose.yml:** added bind mount `./alembic:/app/alembic` so autogen writes land on host. Was required because `/app/alembic/versions` in the image is owned by root while the container runs as `appuser`.
- **`create_type=False` pattern was the fix** for the duplicate `CREATE TYPE` bug in the original migration — but was abandoned when user moved to autogen.

## Failed Approaches

- **Original `0001_initial.py` → `DuplicateObjectError`:** the migration had both `taskstatus.create(op.get_bind())` *and* an `sa.Enum(...)` column without `create_type=False`, so SQLAlchemy emitted `CREATE TYPE taskstatus` twice. Fix: `create_type=False` on the column's `sa.Enum`.
- **First autogen attempt → `FileNotFoundError: alembic/script.py.mako`:** the repo was missing the revision template. Fix: added `alembic/script.py.mako` (standard Alembic template).
- **Second autogen attempt → `PermissionError` writing the generated file:** container runs as `appuser`, `/app/alembic` is baked from root-owned COPY. Fix: bind-mounted `./alembic` in docker-compose + ran `alembic revision` with `-u root`.

## Active Constraints

- **Container user:** `appuser`, not root. `alembic revision --autogenerate` must run with `-u root` (or fix permissions differently).
- **Bind mount required:** don't remove the `./alembic:/app/alembic` mount — autogen would break again.
- **Enum value case mismatch:** DB now stores `'TODO'`, `'IN_PROGRESS'`, `'DONE'` (uppercase, from member names). Pydantic/StrEnum uses lowercase values. Round-trip via SQLAlchemy works; raw SQL or BI tools see uppercase. If this bothers anyone, either re-declare the ORM column with `values_callable=lambda e: [m.value for m in e]` and regenerate, or hand-patch the migration to use lowercase strings.
- **Migration downgrade leaks the enum type:** autogen did not add `DROP TYPE taskstatus` to `downgrade()`. Next `upgrade()` after a downgrade will crash with `DuplicateObjectError`. Patch if downgrade ever matters.
- **No `uuid-ossp` extension, no `server_default` for UUIDs or `is_active`:** autogen stripped these. ORM-driven writes work because the Python layer supplies defaults; raw-SQL inserts would need explicit values.
- **User instruction — never commit or push without approval.** Always pause for review before `git commit` / `git push`.
